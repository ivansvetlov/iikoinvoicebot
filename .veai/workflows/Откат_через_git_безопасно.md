## Зачем
Чтобы можно было “в 1–2 команды” вернуться к стабильной версии, не теряя текущую работу.

## Рекомендованный процесс

### 1) Всегда работаем в ветке
```bash
git checkout -b feature/llm-guards
```

### 2) Маленькие коммиты
Коммить по смыслу: лимиты, детектор, бот-ответы отдельно.

### 3) Тег на стабильной версии
Перед рисковыми изменениями:
```bash
git tag stable-2026-03-08
git push origin stable-2026-03-08
```

### 4) Быстрый откат к тегу (локально)
Если нужно *быстро вернуть как было*:
```bash
git checkout stable-2026-03-08
```

### 5) Правильный откат на main (revert, а не reset)
Если изменения уже в main и нужно откатить без переписывания истории:
```bash
git revert <hash>
```

## Практика для этого проекта
- Держать "рабочий" main
- Любой эксперимент с пайплайном/LLM — отдельной веткой
- Перед запуском в прод/перед большой серией правок — ставить тег

## Команда остановки
Остановись и убедись, что понимаешь разницу `revert` vs `reset`.
