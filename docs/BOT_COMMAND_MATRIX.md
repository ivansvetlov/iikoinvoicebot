# Проверка команд бота: матрица состояний и комбинации (PRE/POST)

> Этот документ нужен, чтобы быстро находить баги в командах и их комбинациях.
> Мы тестируем не «красивый happy path», а поведение при реальных ошибках пользователя.

## PRE: как тестируем (план)

### Состояния пользователя (state machine)
У одного и того же пользователя бот может находиться в одном из состояний:

1) **NO_AUTH** — нет сохранённых iiko-кредов.
2) **AUTH_LOGIN** — после `/start`, ждём логин.
3) **AUTH_PASSWORD** — ждём пароль.
4) **READY** — авторизован, обычная работа.
5) **SPLIT_ON** — пользователь включил `/split`, собираем части до `/done`.
6) **PENDING_FILES** — пользователь отправил файлы подряд (не split), ждём выбора «Обработать» (если 1 файл) или «Объединить».
7) **MEDIA_GROUP** — пользователь отправил альбом (media group), сбор 2 секунды, затем предлагается «Объединить».

### Что проверяем для каждой команды
Для каждой команды проверяем:
- **ожидаемый ответ** пользователю (человеческий текст)
- **изменение состояния** (если должно быть)
- **нет ли конфликтов** с другими состояниями (например /mode во время AUTH_PASSWORD)

### Минимальный набор сценариев (ручной прогон в Telegram)
1) `/start` → логин → пароль → готово.
2) `/mode` в READY (показывает текущий режим).
3) `/modefast` и `/modeaccurate` (в текущей реализации это НЕ отдельные команды, а варианты текста для `/mode`, см. POST).
4) `/split` → отправить 2 фото → кнопка «Завершить».
5) `/split` → `/cancel`.
6) Отправить 2–3 файла подряд без /split → выбрать «Объединить».
7) Отправить 1 файл → нажать «Обработать».
9) Альбом (3 фото в одном сообщении) → получить кнопки «Объединить / Добавить ещё».
10) В каждом сценарии зафиксировать «Код заявки» и прогнать `scripts/diagnose_request.py`.

## POST: что уже нашли (по коду + по ручному прогону)

### Команды, видимые пользователю
В списке команд Telegram показываем только:
- `/start`

`/mode`, `/split`, `/done`, `/cancel` остаются рабочими, но скрыты (резерв, если кнопки недоступны).

### Выбор режима PDF
При получении PDF бот предлагает inline‑кнопки выбора режима (fast/accurate) перед обработкой.

### Баг 1 — команды /modefast и /modeaccurate не зарегистрированы как команды
**Факт (код):** в `_register_handlers()` зарегистрирована только команда `/mode`:
- `self.dp.message.register(self.set_mode, Command("mode"))`

Но в `set_mode()` есть поддержка строк `/modefast` и `/modeaccurate` через `text.startswith("/modefast")` и `text.startswith("/modeaccurate")`.

**Симптом (ручной тест):**
- `/mode` (нажатие на кнопку команды) → бот отвечает общим текстом «принимаю фото/PDF/DOCX…».
- `/modeaccurate` → аналогично.

**Почему это важно:** пользователь ожидает, что команда “режим” работает, а сейчас он попадает в общий обработчик текста.

**Предлагаемое решение (UX проще для людей):**
- оставить одну команду `/mode`, которая показывает текущий режим и даёт inline-кнопки: `fast` / `accurate`.
- дополнительно поддержать текстовый формат: `/mode fast` и `/mode accurate`.
- (опционально) зарегистрировать `/modefast` и `/modeaccurate` как алиасы.

---

### Баг 2 — /split + альбом: «фото добавлено» приходит 3 раза
**Симптом:**
- `/split`
- отправить 3 фото одним альбомом
- бот присылает 3 одинаковых сообщения «Фото добавлено в split» (по одному на каждую часть)

**Почему плохо:** это спамит чат и не даёт понятной “сводки”.

**Предлагаемое решение:**
- в split-режиме агрегировать media group так же, как мы агрегируем обычный batch: одно сообщение “Добавлено 3 фото. Нажмите /done”.

---

### Баг 3 — /done не идемпотентен (можно нажать дважды и отправить две одинаковые задачи)
**Доказательство по логам:**
в `logs/mailbox/6106711925.jsonl` видно два события `split_finish_requested` подряд и два `backend_batch_sending`, что даёт два request_id:
- `225600_911`
- `225601_957`

**Почему плохо:** пользователь получает 2 “кода заявки”, хотя делал одно действие.

**Предлагаемое решение:**
- сделать `/done` идемпотентным: если уже "в процессе отправки" или буфер пуст — отвечать “Уже отправлено” / “Нет частей для отправки”.
- после первого `/done` сразу очищать split-буфер и ставить флаг `split_finishing=true`.

---

### Баг 4 — PENDING_FILES: лишние таймеры и дублирующие сообщения
**Симптом (раньше):**
- появлялись несколько сообщений с кнопками;
- таймер авто-обработки приводил к путанице.

**Технический факт (лог):** есть исключение в `_send_mode_keyboard`:
- `Bad Request: message is not modified` — значит бот пытается редактировать одно и то же сообщение одинаковым текстом/markup.

**Актуальное решение:**
- один файл → кнопка «Обработать» (явный выбор);
- 2+ файлов → кнопка «Объединить»;
- без таймеров, одно “живое” сообщение.

---

### Баг 5 — дедупликация: «этот файл уже был отправлен» мешает повторить отправку, если пользователь не получил результат
**Симптом:** после `/start` и повторной отправки тех же файлов бот отвечает “файл уже был отправлен”.

**Почему плохо:** пользователь не понимает, обрабатывается ли файл, завершилось ли, и что делать если “пропал ответ”.

**Предлагаемое решение:**
- не молча “скипать”, а отвечать: “Этот файл уже обрабатывается. Код заявки: XXXX” (если есть активная задача).
- или делать дедупликацию только на короткий TTL и только для “точно дублей” в течение, например, 30–60 секунд.

---

### Баг 6 — устаревшие кнопки остаются активными
**Симптом:** после таймаута/обработки сообщения с кнопками “объединить/раздельно” всё ещё можно нажимать, хотя pending уже пуст.

**Актуальное решение:**
- после обработки pending используем одно статусное сообщение и редактируем его.

### Баг 7 — старые файлы попадают в новый сценарий
**Симптом:**
- пользователь нажал `/start`, но в pending/split остались файлы из прошлой сессии.

**Актуальное решение:**
- `/start` очищает pending/split буферы пользователя.

## Текущее UX‑поведение (2026-03-11)
- В списке команд Telegram отображается только `/start`. Команды `/mode`, `/split`, `/done`, `/cancel` и алиасы `/modefast` `/modeaccurate` остаются рабочими, но скрыты.
- `/start` очищает pending/split буферы, чтобы не тянуть старые файлы.
- Один файл: сообщение «Файл получен. Что делаем?» + кнопки `Обработать` / `Добавить ещё`.
- 2+ файла (pending): сообщение «Получено файлов: N. Выберите режим» + кнопки `Объединить` / `Добавить ещё`. Старое prompt‑сообщение удаляется.
- Media group (альбом): поведение как у pending, с тем же выбором режима для 2+ файлов.
- Split‑режим: единое сообщение с количеством файлов + кнопки `Отменить` / `Завершить` / `Добавить ещё`; старое сообщение удаляется, кнопки не «висят».
- PDF: при получении PDF показываются inline‑кнопки выбора режима `fast/accurate/продолжить`; `/mode` тоже работает.
- Итог успешного распознавания: форматированный ответ по накладной + inline‑кнопки `Редактировать` / `Отправить в iiko` / `Отмена` и отображение кода заявки.
