# Проверка команд бота: матрица состояний и комбинации (PRE/POST)

> Этот документ нужен, чтобы быстро находить баги в командах и их комбинациях.
> Мы тестируем не «красивый happy path», а поведение при реальных ошибках пользователя.

## PRE: как тестируем (план)

### Состояния пользователя (state machine)
У одного и того же пользователя бот может находиться в одном из состояний:

1) **NO_AUTH** — нет сохранённых iiko-кредов.
2) **AUTH_LOGIN** — после `/start`, ждём логин.
3) **AUTH_PASSWORD** — ждём пароль.
4) **READY** — авторизован, обычная работа.
5) **SPLIT_ON** — пользователь включил `/split`, собираем части до `/done`.
6) **PENDING_FILES** — пользователь отправил 2+ файлов подряд (не split), ждём выбора «Объединить/Раздельно» или автотаймер.
7) **MEDIA_GROUP** — пользователь отправил альбом (media group), сбор 2 секунды, затем batch в backend.

### Что проверяем для каждой команды
Для каждой команды проверяем:
- **ожидаемый ответ** пользователю (человеческий текст)
- **изменение состояния** (если должно быть)
- **нет ли конфликтов** с другими состояниями (например /mode во время AUTH_PASSWORD)

### Минимальный набор сценариев (ручной прогон в Telegram)
1) `/start` → логин → пароль → готово.
2) `/mode` в READY (показывает текущий режим).
3) `/modefast` и `/modeaccurate` (в текущей реализации это НЕ отдельные команды, а варианты текста для `/mode`, см. POST).
4) `/split` → отправить 2 фото → `/done`.
5) `/split` → `/cancel`.
6) Отправить 2–3 файла подряд без /split → выбрать «Объединить».
7) То же → выбрать «Раздельно».
8) То же → ничего не выбирать, дождаться автотаймера.
9) Альбом (3 фото в одном сообщении) → убедиться, что уходит в `/process-batch`.
10) В каждом сценарии зафиксировать «Код заявки» и прогнать `scripts/diagnose_request.py`.

## POST: что уже нашли (статический аудит кода)

### Баг 1 — команды /modefast и /modeaccurate не зарегистрированы как команды
**Факт:** в `_register_handlers()` зарегистрирована только команда `/mode`:
- `self.dp.message.register(self.set_mode, Command("mode"))`

Но в `set_mode()` есть поддержка строк `/modefast` и `/modeaccurate` через `text.startswith("/modefast")` и `text.startswith("/modeaccurate")`.

**Почему это важно:** в aiogram фильтр `Command("mode")` обычно ловит только `/mode ...`. 
То есть `/modefast` и `/modeaccurate` могут не попадать в `set_mode`, а уйти в общий текстовый обработчик (`on_text`) или «unsupported».

**Как проверить быстро:** отправь в чат:
- `/modefast`
- `/modeaccurate`

Ожидаемое (как задумано): «Готово. Режим PDF: fast/accurate.»
Если вместо этого бот отвечает «Я принимаю фото/PDF/DOCX…» — значит баг подтвердился.

**Потенциальный фикс:** зарегистрировать `Command("modefast")` и `Command("modeaccurate")` отдельными хендлерами или заменить UX на `/mode fast`.

### Баг 2 — разные «каналы ввода» конкурируют
В `manager.py` порядок хендлеров такой:
- `on_text` с фильтром `F.text` — ловит почти весь текст.
- команды `/start`, `/mode`, `/split`, `/done`, `/cancel`, `/multi` ловятся раньше.

Риск: «псевдо-команды» (типа `/modefast`, если не зарегистрировано) уйдут в `on_text` и дадут неожиданный ответ.

## Как документировать результаты тестирования
После прогона сценариев ниже дополняем этот документ в секции POST:
- какие команды дали неожиданный ответ
- какой state был
- как воспроизвести (3 шага)
- какой request code получился

Рекомендуемый формат записи:

**Кейс:** `/modefast` в READY
- Ожидаемо: режим fast
- Фактически: сообщение «Я принимаю фото…»
- Код заявки: (если был)
- Вердикт: баг/не баг
