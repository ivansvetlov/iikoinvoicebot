# AGENTS.md — правила сопровождения проекта

> Этот файл — основной источник правил для агентов/разработчиков.

## Каноническая документация
- Канонические Markdown-доки живут в `docs/`.
- В корне проекта из Markdown оставляем только `README.md`.
- Проверенные команды запуска/диагностики — в `docs/DEBUG.md`.

## Когда обновлять документацию
Если меняете поведение, структуру, конфигурацию или операционные процедуры, обновляйте:

**Всегда:**
- `docs/AGENT_HANDOFF.md` — короткий пункт о том, что изменилось и где.

**По необходимости:**
- `docs/README.md` — если влияет на установку/запуск/конфиг.
- `docs/TODO.md` — если это меняет план работ.
- `docs/ARCHITECTURE.md` — если меняется архитектура/потоки.
- `docs/DEV_SETUP.md` — если меняется локальный запуск/операционные шаги.
- `docs/BOT_COMMAND_MATRIX.md` — если меняются команды/UX/кнопки.
- `docs/TESTCASES.md` — если меняются сценарии и ожидания QA.
- `.veai/workflows/*.md` — только если вы реально используете Veai и меняли шаги.

## Минимум, что нужно зафиксировать
- Какие файлы изменены/добавлены/удалены (пути).
- Что изменилось в поведении (1–3 пункта).
- Как быстро проверить (команда или 1 ручной шаг), если применимо.

## Нельзя
- Не записывать секреты в Markdown.
- Не коммитить `.env`, `logs/`, `data/`, `tmp/`, `.venv/`.

## Mailbox Rule (Strict)
- Любые исполняемые блоки команд для пользователя (Termux/Windows/SSH) агент публикует только через mailbox-канал (`ops/mailbox/for_termux.md` через `wmailbox ...`/`wpaste` flow).
- В чат агент пишет только объяснение, статус и результаты; не публикует там блоки команд для копирования/запуска.
- Если нужен набор команд, агент сначала кладёт его в mailbox, затем в чате сообщает, что блок готов к получению через `wpaste`.
